GoPin Tool-less version pinning for Go

Ever used a package which broke backwards compatibility? 

import (
    "github.com/surma/httptools"
)

↓

Pin it to the commit you have developed against!

import (
    "gopin.org/github.com/surma/httptools/ef4a90a9c9a90152e154fa8ba46cc8858f3476ec"
)

That’s all.

Warning: GoPin is neither well-tested nor feature complete.

Important facts

The good:

  • GoPin is open source and MIT licensed
  • No local tools are needed
  • Your projects remain go get-able
  • No additional strain on the maintainer’s or the user’s side

The bad:

  • Only Github is supported (others soon™)
  • Only full commit hashes are supported (tags soon™)

The ugly:

  • Using GoPin will result in having a new clone in your $GOPATH for each pinned import
  • GoPin will not work with private repositories (but neither will go get)
  • Prone to force-pushes (what isn’t?)

How does it work?

GoPin acts as a man in the middle between the go tool and Github. It parses the data stream while the repository is being cloned. When the position of the master branch is being transmitted, GoPin injects the given hash.

In contrast to github.com, gopin.org is not natively supported by the go tool. The custom remote import feature is used to make the go tool know where to find the repository for any given import path.
